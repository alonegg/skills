# 计算机科学类 学科配置

> 本文件是 paper-review 技能的学科叠加层。它不替换通用规则，而是在通用规则基础上增加计算机科学类学科特有的审查维度、调整权重、补充常见问题模式。

## 适用范围

- **覆盖学科**：计算机科学与技术、软件工程、数据科学与大数据技术、人工智能、信息安全、物联网工程
- **典型论文类型**：
  - **系统开发类**：设计并实现一个软件系统（Web应用、移动端应用、嵌入式系统、分布式平台等）
  - **算法/模型类**：提出或改进算法、深度学习模型、优化方法
  - **数据分析类**：基于数据集进行挖掘、分析、可视化，得出结论
  - **综述/对比类**：对某技术领域进行系统性调研和对比评估
- **识别线索**：论文中出现系统架构图、类图/ER图、算法伪代码、实验环境配置表、代码仓库链接、API接口描述、数据集描述、模型训练参数等

## 结构期望

### 标准章节结构

本学科论文结构因论文类型而异，审查时应先判断论文属于哪种类型，再对照对应结构：

**系统开发类**（最常见）：
```
第1章 绪论（研究背景、目的、意义、论文结构）
第2章 相关技术介绍（技术选型及理由）
第3章 需求分析（功能需求、非功能需求、用例分析）
第4章 系统设计（总体架构、模块设计、数据库设计、接口设计）
第5章 系统实现（关键模块实现、核心代码逻辑说明）
第6章 系统测试（测试方案、测试用例、测试结果与分析）
第7章 总结与展望
```

**算法/模型类**：
```
第1章 绪论（研究背景、问题定义、研究贡献）
第2章 相关工作（现有方法综述、不足分析）
第3章 问题定义与形式化（数学符号、问题建模）
第4章 方法设计（算法/模型描述、理论分析）
第5章 实验评估（实验设置、对比实验、消融实验、结果分析）
第6章 总结与展望
```

**数据分析类**：
```
第1章 绪论
第2章 文献综述与理论基础
第3章 数据来源与预处理方法
第4章 实验设计与分析
第5章 结果与讨论
第6章 总结与展望
```

**综述/对比类**：
```
第1章 绪论
第2章 背景知识与定义
第3章 分类框架（提出对比维度/分类标准）
第4章 系统性对比分析
第5章 讨论与未来方向
第6章 总结
```

### 必须有的章节

- **绪论**：必须包含明确的问题陈述（本文要解决什么问题）和研究贡献概述（本文做了什么），不能只有空泛的"意义"
- **相关工作/技术**：必须存在，且不能是教科书式的技术罗列（见常见问题）
- **核心工作章节**：系统开发类必须有设计+实现两个独立章节；算法类必须有方法设计+实验两个独立章节
- **测试/实验/评估**：任何类型的论文都必须有验证环节，系统开发类不能跳过测试直接结论
- **总结与展望**：必须基于本文实际工作进行总结，展望必须与本文方向相关

### 可选/特殊章节

- **性能优化章节**：当系统有高性能/高并发需求时需要单独讨论
- **安全分析章节**：信息安全方向论文必须有威胁模型和安全性分析
- **部署与运维**：涉及实际上线的系统可增加部署方案讨论
- **伦理与隐私**：涉及用户数据、人脸识别等敏感领域时需要讨论

### 结构红线

- **无测试/无实验的"完整系统"**：声称实现了系统但没有任何测试章节——结构不完整，S级问题
- **系统设计与实现混为一谈**：将架构图和代码截图放在同一章节，没有设计层到实现层的分离
- **相关工作章节缺失**：直接在绪论中用一段话带过所有相关研究
- **结论章节照抄摘要**：总结与展望内容与摘要完全重复，没有增量信息

## 专项规则权重调整

基于通用6大专项规则，调整本学科的审查权重：

| 专项规则 | 权重调整 | 理由 |
|---|---|---|
| intro | +1 | 计算机类论文绪论中必须明确技术问题和贡献点，空泛的"信息化建设"式开头是普遍顽疾 |
| literature | +1 | 需严格区分"技术介绍"和"文献综述"，防止教科书式堆砌 |
| method | +2 | 方法/设计是计算机类论文的核心，必须有技术深度，防止沦为框架使用说明 |
| analysis | +2 | 实验/测试的严谨性是计算机论文质量的关键区分度，需重点审查 |
| conclusion | 0 | 无特殊调整 |
| fact | -1 | 计算机类论文中纯事实性段落较少，更多是技术描述和设计决策 |

## 方法审查补充维度

通用 method 规则之外，本学科需额外检查：

### 系统开发类论文

1. **技术贡献度审查**：系统是否有超出简单CRUD（增删改查）的技术含量？审查要点：是否存在算法逻辑、并发处理、缓存策略、消息队列、分布式架构等技术决策？如果一个系统完全可以用低代码平台搭建，则技术贡献不足
2. **架构设计合理性**：架构图是否反映了系统的实际复杂性？各模块之间的交互关系是否清晰？是否存在架构图与实际实现不一致的情况（如架构图画了微服务但实现是单体应用）
3. **技术选型论证**：是否说明了为什么选择这套技术栈？选择Spring Boot而非Django、选择MySQL而非MongoDB——需要有基于需求的技术论证，而非"因为流行"
4. **数据库设计规范性**：ER图是否规范？表结构设计是否遵循范式（或有合理的反范式理由）？是否存在明显的冗余或设计缺陷
5. **接口设计完整性**：系统对外接口是否有完整定义？RESTful API是否遵循规范？是否考虑了鉴权、限流、错误处理

### 算法/模型类论文

6. **问题形式化**：问题是否有严格的数学定义（输入、输出、约束、优化目标）？不能只用自然语言描述问题
7. **算法正确性论证**：是否提供了正确性证明或至少正确性论述？贪心算法需要证明贪心选择性质，DP需要证明最优子结构
8. **复杂度分析**：时间复杂度和空间复杂度是否给出？是否与实验中的运行时间一致？
9. **基线对比完整性**：是否与当前SOTA方法对比？是否包含经典基线？不能只和自己的消融版本比
10. **消融实验**：如果方法有多个组件/创新点，是否通过消融实验验证了每个组件的贡献？

### 机器学习/AI类论文

11. **数据集规范性**：数据来源是否清晰？数据集规模是否足够？数据预处理步骤是否完整？是否存在数据泄露（训练集信息渗入测试集）
12. **训练/测试划分**：划分比例和策略是否合理？是否使用了交叉验证？是否报告了多次运行的均值和标准差？
13. **超参数说明**：学习率、batch size、epoch数、优化器等关键超参数是否完整列出？是否说明了调参策略（网格搜索/随机搜索/贝叶斯优化）？
14. **评估指标选择**：指标是否适合任务（分类用Accuracy/F1/AUC、检测用mAP、生成用BLEU/ROUGE等）？不平衡数据集是否避免了只用Accuracy？

### 信息安全类论文

15. **威胁模型定义**：是否明确定义了攻击者的能力假设、攻击目标、攻击面？
16. **安全性论证**：安全机制是否有形式化的安全性证明或安全性分析？是否讨论了已知攻击向量？
17. **攻防实验**：是否设计了攻击实验来验证防御方案的有效性？是否测试了对抗性场景？

### 物联网类论文

18. **硬件环境说明**：硬件平台型号、传感器参数、通信协议是否清晰？
19. **实时性与资源约束**：是否讨论了时延、功耗、内存占用等物联网特有约束？
20. **端到端验证**：是否在真实设备或高仿真环境上进行了验证，而非仅仿真？

## 证据与论证标准补充

通用 analysis/fact 规则之外，本学科的证据标准：

### 可接受的证据类型

- **实验数据**：在明确实验环境下产生的可量化结果（响应时间、吞吐量、准确率、F1值等），需注明实验环境配置（CPU型号、内存、GPU型号、操作系统、语言/框架版本）
- **对比实验**：与基线方法在相同条件下的量化对比，需确保对比公平性（相同数据集、相同评估协议、相同硬件）
- **测试报告**：单元测试覆盖率、集成测试通过率、性能压测结果（如JMeter/Locust报告），需附测试用例设计依据
- **复杂度证明**：时间/空间复杂度的数学推导，可作为算法性能论证的证据
- **用户测试**：可用性评估（SUS量表、任务完成率），仅作为辅助证据，不能替代技术测试
- **形式化验证**：安全协议的形式化证明（如BAN逻辑、ProVerif分析结果）
- **开源基准测试**：使用公认benchmark的评测结果（如ImageNet、GLUE、TPC-C）

### 证据红线

- **无对比的性能声明**："系统响应时间在200ms以内"——与什么对比？在什么负载下？绝对数字无意义，必须有参照系
- **截图当证据**：用系统运行截图代替测试数据，"系统能正常运行"不是有效证据
- **Accuracy幻觉**："模型准确率达到95%"——在什么数据集上？与什么基线比？95%是否有意义（如果正负样本比9:1，Accuracy 90%就是全猜多数类）？
- **孤立指标**：只报告对自己有利的指标，回避其他标准指标（如只报Accuracy不报F1/Recall）
- **无随机种子**：深度学习实验未固定随机种子，无法判断结果是否可复现
- **训练集上的结果**：用训练集指标代替测试集指标来声称模型有效
- **不公平对比**：自己的方法用最优超参数，基线方法用默认参数或未调优
- **P-hacking**：大量尝试后只报告最好的一次结果，不报告均值和方差

### 论证模式期望

- **系统开发类**：需求→设计决策→实现→测试验证的因果链完整。每个设计决策应有需求依据，每个实现应有对应的测试验证
- **算法类**：问题定义→方法设计→理论分析→实验验证的演绎-实证结合。理论与实验应互相印证
- **数据分析类**：假设→方法→结果→解释的实证研究范式。结果的解释不能超出数据支撑的范围
- **综述类**：分类维度→系统对比→差距识别→趋势判断的归纳论证。对比维度必须合理且完整

## 常见问题模式

### S级（论文是否成立）

1. **系统"开发"无技术实质**：
   - 表现：论文声称设计实现了一个系统，但全文只有框架配置和CRUD操作，系统设计章节仅包含从教程复制的架构图，无任何技术决策讨论
   - 影响：论文没有本科毕业设计应有的技术贡献，不构成学术工作
   - 修改方向：识别系统中真正有技术含量的模块（如推荐算法、权限控制模型、并发处理），深入设计和讨论；或更换选题

2. **模型/算法无基线对比**：
   - 表现："本文提出的模型准确率为95%"，全文无任何对比实验，无基线方法，读者无法判断95%的含义
   - 影响：核心实验结论无法成立，无法证明方法的有效性
   - 修改方向：补充至少2-3个基线方法（经典方法+近年SOTA），在相同数据集和实验设置下进行公平对比

3. **无测试/无验证**：
   - 表现：系统"实现完成"后直接跳到结论，没有测试章节；或测试章节仅有2-3张系统截图
   - 影响：无法证明系统确实能正常工作，论文的核心成果未被验证
   - 修改方向：补充完整的测试方案——功能测试（覆盖核心用例）、性能测试（响应时间/并发量）、兼容性测试

4. **代码架构抄袭**：
   - 表现：系统架构、数据库设计、页面布局与某个公开教程项目或GitHub开源项目高度一致，仅修改了业务名称
   - 影响：学术诚信问题，论文不构成独立工作
   - 修改方向：如基于开源项目，必须明确标注来源并说明自己的扩展贡献；核心功能必须有原创技术工作

5. **数据泄露**：
   - 表现：ML论文中测试集信息渗入训练过程（如用全量数据做特征工程后再划分、测试集参与了归一化计算）
   - 影响：实验结果全部无效，所有性能数据不可信
   - 修改方向：严格时间序列划分或随机划分后，所有预处理步骤仅在训练集上fit，再transform到测试集

### A级（严重影响质量）

1. **"相关技术"章节教科书化**：
   - 表现：用2-3页解释"MySQL是一种关系型数据库管理系统"、"Spring Boot是基于Spring框架的快速开发工具"、"Vue.js是一种渐进式JavaScript框架"，内容完全是官方文档或教材的复述
   - 影响：浪费论文篇幅，暴露作者对"相关工作"与"技术介绍"的混淆，评审观感极差
   - 修改方向：相关技术章节应聚焦于技术选型论证（为什么选这个而不选那个）和与本系统相关的关键技术点，而非通用教程

2. **架构图与实现脱节**：
   - 表现：系统设计章节画了微服务架构图（多个独立服务+消息队列+API网关），但实现章节展示的是单体Spring Boot应用
   - 影响：设计与实现不一致，读者无法信任论文的技术描述
   - 修改方向：架构图必须真实反映已实现的系统，未实现的部分如需展示，应明确标注为"设计目标"

3. **只测试正常路径**：
   - 表现：测试用例全部是正常输入→正常输出，没有边界值测试、异常输入测试、并发测试、安全测试
   - 影响：测试不能有效证明系统的健壮性
   - 修改方向：补充边界条件测试、异常输入处理、错误恢复机制验证；系统类论文至少覆盖每个模块的异常场景

4. **代码替代设计**：
   - 表现：系统实现章节大段粘贴代码（controller代码、SQL语句、配置文件），没有对设计思路和技术决策的讨论
   - 影响：论文变成代码报告而非学术论文，缺乏抽象和分析
   - 修改方向：代码展示应精选关键片段（核心算法、关键数据结构），重点说明设计决策（为什么这样做、考虑了哪些替代方案、有什么权衡）

5. **实验环境信息缺失**：
   - 表现：直接报告实验结果，但未说明硬件配置、软件版本、数据集划分方式、随机种子
   - 影响：实验不可复现，结果的可信度降低
   - 修改方向：以表格形式完整列出实验环境（CPU/GPU型号、内存、OS、Python/Java版本、框架版本、随机种子、数据集划分比例）

6. **不讨论局限性**：
   - 表现：全文只有正面结论，没有讨论方法/系统的局限性、适用条件、失败案例
   - 影响：论文缺乏学术诚实度，评审会质疑作者是否真正理解自己的工作
   - 修改方向：在实验/讨论章节坦诚分析方法的不足、系统的已知缺陷、性能的瓶颈所在

### B级（影响说服力）

1. **图表不规范**：
   - 表现：架构图使用PPT随意绘制（圆角矩形+箭头，无标准符号）、数据库设计没有正式ER图、流程图不符合标准、实验图表没有图例或坐标轴标签
   - 影响：降低论文的专业性和可读性
   - 修改方向：UML图使用标准符号，ER图使用Chen或Crow's Foot表示法，流程图符合ISO 5807标准，所有实验图表必须有标题、图例、坐标轴标签和单位

2. **参考文献质量低**：
   - 表现：参考文献以CSDN博客、百度百科、框架官方文档、教材为主，缺少学术论文
   - 影响：论文缺乏学术支撑，文献基础薄弱
   - 修改方向：补充高质量学术论文（见文献综述要求），博客和文档可以引用但不能作为主要参考文献

3. **变量命名不一致**：
   - 表现：同一个概念在不同章节使用不同名称（如"用户模块"/"用户管理子系统"/"用户服务"交替出现）
   - 影响：增加读者理解负担，暴露写作不严谨
   - 修改方向：统一术语，首次出现时给出明确定义，全文一致使用

4. **实验分析浅尝辄止**：
   - 表现：列出实验结果表格后仅用"由表可知本文方法优于其他方法"一句话总结，不分析为什么优、在哪些情况下优、什么时候不优
   - 影响：丧失了实验分析的深度，无法体现对结果的理解
   - 修改方向：对每组实验结果进行深入分析——性能差异的原因、不同场景下的表现差异、失败案例的分析

5. **系统截图过多**：
   - 表现：实现章节包含大量系统运行截图（登录界面、列表页面、表单页面），每个截图占半页
   - 影响：论文篇幅虚高但信息密度低
   - 修改方向：截图应精选能展示核心功能或技术特点的界面，普通CRUD页面无需截图

## 文献综述特殊要求

### 文献来源优先级

1. **顶级学术会议论文**：
   - 人工智能/机器学习：NeurIPS、ICML、ICLR、AAAI、IJCAI
   - 计算机视觉：CVPR、ICCV、ECCV
   - 自然语言处理：ACL、EMNLP、NAACL
   - 数据库/数据挖掘：SIGMOD、VLDB、KDD、ICDE
   - 系统/网络：OSDI、SOSP、NSDI、SIGCOMM
   - 安全：IEEE S&P、CCS、USENIX Security、NDSS
   - 软件工程：ICSE、FSE、ASE
   - 参考CCF推荐目录（A/B/C类）

2. **顶级期刊论文**：IEEE TPAMI、JMLR、ACM Computing Surveys、IEEE TDSC等

3. **arXiv预印本**：可引用最新研究进展，但需注明为预印本，不能作为唯一文献来源

4. **技术报告与白皮书**：来自Google、Microsoft、Meta等研究机构的技术报告可以引用

5. **框架文档与标准**：作为技术说明的依据可以引用，但不计入"学术参考文献"

### 文献数量底线

- 系统开发类：至少10篇学术论文（不含框架文档和教材）
- 算法/模型类：至少20篇学术论文，其中近3年文献占比不低于40%
- 综述类：至少30篇学术论文，需体现对该领域的系统性覆盖

### 综述组织方式

- **禁止**：逐篇罗列式综述（"张三[1]做了…，李四[2]做了…，王五[3]做了…"），这不是综述，是文献列表
- **要求**：按技术路线、方法类别、或发展阶段组织，形成分析框架。每个类别需有优缺点总结，最终引出本文工作的定位
- **必须有**：综述末尾的"总结与本文定位"段落，明确说明现有工作的不足和本文的切入点

## 术语与规范

### 术语使用规范

- **中英文对照**：首次出现的英文术语需给出中文译名或保留英文原文并加括号说明，之后全文统一使用一种形式。例如："卷积神经网络（Convolutional Neural Network, CNN）"，此后统一使用"CNN"或"卷积神经网络"
- **缩略语管理**：所有缩略语首次出现时必须给出全称，建议在论文开头设置缩略语表
- **避免口语化**：不使用"搞一个系统"、"跑一下模型"等口语表达，使用"设计实现系统"、"执行模型训练"
- **技术术语准确性**：区分"算法"与"模型"、"精度"与"准确率"、"框架"与"库"等易混淆术语

### 引用格式

- 遵循GB/T 7714-2015标准（中文学位论文通用要求）
- 会议论文引用需包含：作者、论文标题、会议名称（缩写）、年份、页码
- 在线资源需注明访问日期
- 代码仓库引用需注明commit hash或版本号

### 图表规范

- **UML图**：使用标准UML 2.x符号，推荐使用draw.io、StarUML、PlantUML等专业工具绘制，不接受随手画的方框图充当类图
- **架构图**：必须标注组件名称、组件间通信协议/方式、数据流方向
- **ER图**：使用标准表示法（Chen或Crow's Foot），标注实体属性、主键、外键、关系类型（1:1、1:N、M:N）
- **流程图**：遵循标准流程图符号（开始/结束椭圆、处理矩形、判断菱形、数据平行四边形）
- **实验结果图**：必须包含标题、坐标轴标签（含单位）、图例、误差线（如适用）；折线图/柱状图的y轴应从合理值开始（不通过截断y轴制造视觉放大效果）
- **代码展示**：使用等宽字体和语法高亮，注释核心逻辑，不超过20行（关键片段而非整文件）

### 公式规范

- 所有公式需编号以便引用
- 公式中的变量在首次出现时需给出定义和含义
- 复杂公式需逐步推导，不要跳步

## 结论与建议特殊要求

### 结论章节要求

- **成果量化**：系统开发类需总结实现了哪些功能模块、测试指标如何；算法类需总结在哪些指标上达到了什么水平、比基线提升了多少
- **禁止空泛总结**："本文设计了一个基于B/S架构的XX管理系统，实现了XX功能，具有一定的实用价值"——这种结论毫无信息量，必须具体
- **技术贡献明确化**：必须明确说明本文的技术贡献点是什么（不是"实现了一个系统"，而是"提出了XX机制解决了XX问题"）
- **局限性坦诚**：必须讨论当前工作的局限性和不足

### 展望部分的边界

- **合理展望**：基于当前工作的已知不足提出改进方向，如"未来将在更大规模数据集上验证"、"将引入XX技术优化XX模块的性能"
- **过度展望红线**：展望的内容与本文工作无关（如论文做的是Web系统，展望里写"未来将探索量子计算在该领域的应用"）；展望的内容过于宏大和空泛（"将为我国信息化建设做出贡献"）
- **合理粒度**：展望应具体到可操作的下一步工作，而非宏观趋势预测

## 学科专属审查维度

### 代码质量与可复现性

审查论文是否满足以下代码相关要求：

- **代码可获取性**：论文是否提供了代码仓库链接？如果声称开源，链接是否有效？
- **依赖说明**：是否提供了requirements.txt、pom.xml或package.json等依赖配置？
- **运行说明**：是否有足够的信息让读者从零开始运行系统/复现实验？
- **数据可获取性**：实验用的数据集是否公开？如果是自采数据，是否提供了数据样例和采集说明？
- **注意**：本科论文不强制要求开源代码，但如果论文中声称了可复现性或引用了代码仓库，则必须验证上述信息的完整性

### 图表质量专项检查

| 图表类型 | 必须包含的要素 | 常见问题 |
|---|---|---|
| 系统架构图 | 组件名、通信方式、数据流向 | 方框+箭头无标注，看不出实际架构 |
| 数据库ER图 | 实体、属性、关系类型、主外键 | 只有表名和字段名的表格，不是ER图 |
| UML类图 | 类名、关键属性/方法、关系类型 | 不规范的方框图冒充类图 |
| 时序图 | 参与者、消息、返回值、生命线 | 仅画了调用关系，没有时序信息 |
| 实验对比图 | 标题、坐标轴标签、单位、图例、误差线 | 无图例、y轴截断、无误差信息 |
| 流程图 | 标准符号、判断条件、分支标注 | 全用矩形，判断条件不清楚 |

### "系统类"论文技术深度评估框架

用于判断一篇系统开发类论文是否具有足够的技术深度：

**深度不足的信号**（多项命中则技术贡献存疑）：
- 系统所有功能都可以用一个框架的脚手架生成
- 全文没有出现任何算法描述（伪代码、状态机、策略模式等）
- 数据库表少于5张且全部是简单的一对多关系
- 系统没有任何异步处理、缓存、消息队列、定时任务等中间件使用
- "核心模块实现"章节只展示了Controller层代码
- 性能需求章节缺失或仅有一句"系统响应时间满足要求"

**深度充足的信号**（至少命中2-3项）：
- 包含自主设计的算法/策略（推荐算法、调度算法、匹配算法等）
- 有明确的性能优化措施并提供了优化前后的对比数据
- 涉及非平凡的系统架构设计（缓存层、消息队列、分布式部署等）并有技术论证
- 包含安全性设计（认证授权方案、数据加密、SQL注入防护等）的深入讨论
- 有完整的测试体系（单元测试+集成测试+性能测试），测试覆盖率有具体数据
- 处理了真实的技术挑战（高并发、大数据量、实时性要求）并有解决方案
