# 学科配置维护规范

本文件是 paper-review 技能的学科配置维护指南。当需要新增、修订或审查学科配置时，参照本规范执行。

---

## 文件版本管理

每个学科配置文件顶部应包含版本元数据：

```markdown
<!-- version: 1.0 | last_reviewed: 2026-03-17 | author: Claude -->
```

- **version**：主版本.次版本。主版本 = 结构性重写；次版本 = 内容增补或修正
- **last_reviewed**：最近一次人工或审阅实践验证的日期
- **author**：编写/最后修订者

每次修改时更新版本号，并在文件末尾的 Changelog 中记录变更。

---

## 新增学科配置

### 前置条件

1. 确认该学科与现有配置无法合并（方法论差异足够大）
2. 有足够的学科知识支撑（至少了解该学科的论文结构、方法标准、常见缺陷）
3. 使用 `_template.md` 作为起点

### 编写原则

1. **写差异，不写通用**：通用规则已在 SKILL.md 和 sub-skills 中，学科配置只写该学科特有的内容
2. **写标准，不写教程**：目标读者是审阅系统（Claude），不是学生。描述"应该检查什么"和"什么算问题"，不需要解释基础概念
3. **写具体，不写空泛**：
   - 差："`方法需要严谨`"
   - 好："`回归模型必须报告R²、F值、各系数t值及显著性水平；样本量低于变量数10倍时应警告过拟合风险`"
4. **写判别条件，不写主观评价**：每个问题模式应包含可操作的判别标准，而非"感觉不够好"
5. **分级要准确**：S级 = 论文不成立，A级 = 严重影响质量。不要通胀——如果什么都是S级，分级就失去意义

### 质量检查清单

新配置完成后，逐项检查：

- [ ] 覆盖学科和识别线索是否明确？能否区分本配置与相邻学科？
- [ ] 结构期望是否列出了该学科的多种论文类型？（本科论文不只一种写法）
- [ ] 方法审查是否包含该学科特有的方法论检查维度？（不能只有通用维度）
- [ ] 常见问题模式是否来自真实的审阅经验/学科认知？（不是凭空编造）
- [ ] S级问题是否真的会导致论文不成立？（避免分级通胀）
- [ ] 文献来源优先级是否列出了该学科的核心期刊/会议？
- [ ] 术语规范是否包含该学科的特有要求？

## 修订现有配置

### 触发条件

1. **审阅实践反馈**：在实际审阅中发现某条规则不准确、过严或过松
2. **用户纠正**：用户指出某个学科判断有误
3. **知识更新**：新的方法、工具、标准成为主流（如大模型进入各学科）
4. **配置缺口**：审阅时发现某个维度配置中没有覆盖

### 修订流程

1. 定位需修改的节（结构/方法/证据/问题模式/文献/术语）
2. 修改内容，保持与模板结构一致
3. 更新文件顶部的 version 和 last_reviewed
4. 在文件末尾 Changelog 中记录：日期、修改内容、修改原因

### 修订原则

- **最小改动**：只改需要改的部分，不借机重写全文
- **保留原始判断的合理性**：如果原规则在大多数场景正确，用"补充例外"而非"替换规则"
- **标注来源**：修正来自用户反馈的，标注"来源：审阅实践反馈"；来自知识更新的，标注"来源：学科知识更新"

## 定期审查

### 频率

- **活跃学科**（使用频率高的配置）：每 6 个月审查一次
- **低频学科**：每 12 个月审查一次
- **全局审查**：当 SKILL.md 主控层发生重大变更时，检查所有配置的兼容性

### 审查内容

1. 结构期望是否仍然反映当前的学科论文实践？
2. 方法标准是否需要纳入新方法？（如新的统计方法、新的研究范式）
3. 常见问题模式是否仍然是"常见"的？是否有新的高频问题需要补充？
4. 文献来源优先级是否需要更新？（新创刊的期刊、停刊的期刊、影响力变化）
5. 是否有审阅实践中积累的反馈尚未整合？

## 用户注入知识的沉淀

当用户在审阅前提供学科专有知识（如学校规范、导师偏好、特殊标准）时：

### 沉淀判断标准

| 用户提供的内容 | 沉淀位置 | 理由 |
|---|---|---|
| 该学科的通用标准/方法要求 | `disciplines/[学科].md` 主文件 | 对所有同学科审阅有价值 |
| 特定学校/院系的格式规范 | 不沉淀到学科配置 | 属于院校层，存入审阅目录即可 |
| 对某条规则的纠正 | `disciplines/[学科].md` 修订 | 提升规则准确性 |
| 一次性的审阅偏好 | 不沉淀 | 仅适用于当次审阅 |

### 沉淀流程

1. 判断用户知识属于哪个层次（学科通用 vs 院校特定 vs 一次性）
2. 如果是学科通用知识，在当次审阅完成后提示：

   > "本次审阅中您提供了关于[具体内容]的学科标准，是否将其纳入学科配置？这将提升后续同学科论文的审阅质量。"

3. 用户确认后，更新对应的学科配置文件，走修订流程

## Changelog 格式

每个学科配置文件末尾应维护 Changelog：

```markdown
---

## Changelog

### v1.0 (2026-03-17)
- 初始版本，覆盖 [学科列表]
- 编写者：Claude
- 知识来源：训练数据 + 学科通识
```
